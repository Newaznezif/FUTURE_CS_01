  1. Manual Testing Findings

Vulnerability 1: Cross-Site Scripting (XSS)

Risk Level: High

Description: During manual testing, a reflected XSS vulnerability was identified in the application’s search functionality. User-supplied input was reflected in the response without proper sanitization or encoding, allowing JavaScript execution in the browser.

The target application for this assessment is OWASP Juice Shop, an intentionally vulnerable web application designed for learning web security testing. The application was deployed locally and accessed via http://localhost:3000. Testing was performed in a controlled environment for educational purposes only.

Steps to Reproduce

  • Navigated to the Juice Shop application
  • Locate the search bar
  • Enter the following payload: <img src="invalid_source" onerror="alert('Successful XSS');"> or <details open ontoggle="alert('XSS')"></details>
  • Submitted the search request: http://localhost:3000/#/search

Result: The JavaScript payload executed successfully, displaying an alert dialog. This confirms the presence of a reflected XSS vulnerability.

Impact: Exploiting this XSS vulnerability could allow an attacker to:

  • Steal session cookies and hijack user sessions
  • Execute actions on behalf of users (CSRF-like impact)
  • Inject and execute arbitrary JavaScript in victim browsers
  • Deface web pages or manipulate content
  • Redirect users to malicious websites
  • Potentially steal credentials or sensitive information

Remediation:

  • Implement proper input validation and output encoding
  • Use frameworks that auto-escape user input
  • Apply Content Security Policy (CSP)
  • Avoid use of innerHTML with user-controlled input
  • Properly encode all user-supplied input before rendering
  • Use secure templating frameworks
  • Implement Content Security Policy (CSP)

The automated scan identified multiple vulnerabilities, including:

  • Content Security Policy (CSP) Header Not Set: A medium-risk vulnerability indicating the application lacks a CSP, which is essential for preventing Cross-Site Scripting (XSS) and data injection attacks.
  • Missing Anti-clickjacking Header: The absence of headers like X-Frame-Options or CSP frame-ancestors, which prevents an attacker from loading the site in an invisible iframe to trick users.

  • X-Content-Type-Options Header Missing: This indicates the browser might "sniff" the content type, potentially executing non-executable files as scripts.
  • Information Disclosure: Several instances were found, including Private IP Disclosure, Timestamp Disclosure, and Suspicious Comments, which could provide attackers with technical details about the backend infrastructure.
  • Server Leaks Version Information: The "Server" HTTP response header is revealing specific software version details, which can be used to research known exploits for that version.
  • Session ID in URL Rewrite: This is a security risk where session identifiers are exposed in the URL, making them susceptible to shoulder surfing or logging in server logs.

Vulnerability 3: Missing Content Security Policy (CSP)

Risk Level: Medium

Confidence: High

ZAP Alert Details

Description: The application does not define a Content Security Policy (CSP) header, leaving it more vulnerable to client-side injection attacks such as Cross-Site Scripting (XSS) and data injection. CSP acts as an additional security layer by specifying approved sources of content that browsers may load, covering JavaScript, CSS, HTML frames, fonts, images, and embedded objects. Without it, attackers can exploit these vectors for data theft, site defacement, or malware distribution.

Impact

  • Increased risk of script injection
  • Weak browser-level protection
  • Increased risk of XSS attacks
  • Data theft or leakage
  • Site defacement
  • Distribution of malicious content

Remediation

  • Implement a strict Content Security Policy header
  • Restrict allowed script sources

Vulnerability 4: Cross-Domain Misconfiguration

Risk Level: Medium

Confidence: Medium

ZAP Alert Details

Description: Web browser data loading may be possible due to a Cross-Origin Resource Sharing (CORS) misconfiguration on the web server. The server is configured with the wildcard Access-Control-Allow-Origin: * header. The CORS misconfiguration on the web server permits cross-domain read requests from arbitrary third-party domains using unauthenticated APIs on this domain.

Impact

  • Unauthorized Data Access: Arbitrary third-party domains can read responses from unauthenticated APIs.
  • Bypassing Security Controls: Attackers can potentially bypass security measures like IP address white-listing to access internal data.
  • Privacy Risks: Sensitive information intended for a specific origin could be exposed to malicious websites.
  • Weakened Same-Origin Policy: The configuration effectively disables the browser's Same-Origin Policy (SOP) for the affected resources.

Remediation

  • Restrict Allowed Origins: Configure the Access-Control-Allow-Origin HTTP header to a more restrictive set of trusted domains rather than using a wildcard.
  • Remove Unnecessary Headers: Remove all CORS headers entirely to allow the web browser to enforce the Same-Origin Policy (SOP) in a more restrictive manner.
  • Secure Unauthenticated Data: Ensure that sensitive data is not available in an unauthenticated manner (e.g., do not rely solely on IP white-listing).

Vulnerability 5: Missing Anti-clickjacking Header

Risk Level: Medium

Confidence: Medium

ZAP Alert Details

Description: The application does not include an anti-clickjacking header, such as X-Frame-Options or a Content Security Policy (CSP) frame-ancestors directive. Without these headers, the browser is not instructed on whether it is permissible to render the page within a <frame>, <iframe>, or <object>. Attackers can use this lack of protection to embed the web application into a transparent layer on a malicious site.

Impact

  • Increased risk of UI Redressing: Attackers can trick users into clicking on buttons or links they did not intend to.
  • Unauthorized Actions: Users may unknowingly perform actions such as deleting accounts, changing passwords, or making purchases.
  • Data Leakage: In some scenarios, sensitive information can be captured through deceptive interface overlays.
  • Weak Browser-Level Protection: The browser lacks instructions to block the page from being framed by untrusted origins.

Remediation

  • Implement X-Frame-Options Header: Configure the server to send the X-Frame-Options header with a value of DENY (no framing allowed) or SAMEORIGIN (framing only allowed by the same site).
  • Utilize CSP frame-ancestors: Use the modern Content-Security-Policy header with the frame-ancestors directive to specify exactly which domains are permitted to embed the page.
  • Consistent Deployment: Ensure these security headers are applied across all pages of the application, not just the login or landing pages.

Vulnerability 6: Session ID in URL Rewrite

Risk Level: Medium

Confidence: High

ZAP Alert Details

Description: URL rewrite is used to track user session IDs. The session ID may be disclosed via the cross-site referer header. The scan identified session identifiers (e.g., sid=fm0WOENAF7xBBauQAAAK) being passed directly within the URL string.

Impact

Passing session identifiers in the URL exposes the application to several security risks:

  • Log Exposure: The session ID might be stored in plain text within browser history or server logs, where it can be discovered by unauthorized individuals.
  • Referer Header Leakage: If a user clicks an external link from the application, the session ID is often sent to the third-party site in the HTTP Referer header.
  • Session Hijacking: An attacker who obtains a valid session ID from logs or intercepted traffic can impersonate the user and gain unauthorized access to their account.
  • Shoulder Surfing: The session ID is visible in the browser's address bar, making it vulnerable to anyone physically near the user's screen.

Remediation

  • Use Cookies for Session Management: Configure the application to store session identifiers only in secure, HTTP-only cookies instead of using URL rewriting.
  • Disable URL Rewriting: Ensure that the web server or application framework is configured to disable the automatic appending of session IDs to URLs (e.g., jsessionid in Java or PHPSESSID in PHP).
  • Implement Secure Referrer Policy: Use the Referrer-Policy HTTP header (e.g., no-referrer or same-origin) to prevent the browser from sending sensitive URL data to external sites.

6. Conclusion

The security assessment of the OWASP Juice Shop application successfully identified several vulnerabilities commonly found in real-world web applications. The most critical issues discovered involved Security Misconfigurations and Client-side Injection flaws, which pose a significant risk to users and application integrity.

This exercise provided hands-on experience in:

  • Web vulnerability SSdiscovery: Identifying systematic issues such as missing security headers and exposed endpoints like /ftp or /api.
  • OWASP Top 10 mapping: Categorizing findings under established industry standards, specifically A05:2021 – Security Misconfiguration.
  • Security documentation and reporting: Translating technical scan data into actionable insights and remediation strategies.
  • Ethical hacking methodologies: Utilizing automated tools like OWASP ZAP to simulate attack vectors and verify weaknesses.

The findings demonstrate the critical importance of secure coding practices, proper input validation, and robust security header configuration. Specifically, the lack of a Content Security Policy (CSP) and the use of Session IDs in URL rewrites highlight the need for a defense-in-depth approach.

In summary, this assessment identified multiple vulnerabilities, including high-risk flaws that could allow for Cross-Site Scripting (XSS) or session hijacking. Both manual and automated testing techniques confirmed the presence of exploitable weaknesses. These issues, if present in a production environment, could result in data compromise and user exploitation. Implementing secure coding practices and proper security controls, such as strict CORS policies and Anti-clickjacking headers, is strongly recommended.
